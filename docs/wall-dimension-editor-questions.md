# Вопросы для реализации редактора размеров стен

Список вопросов, ответы на которые нужны, чтобы приступить к реализации
редактора по документу
[wall-dimension-editor-requirements.md](wall-dimension-editor-requirements.md).

**Формат:** под каждым вопросом строка **Ответ:** содержит принятое решение
с кратким обоснованием.

---

## 1. Модель данных и совместимость

1.1. **Совместимость с текущим `plan.json`.**
Сейчас стена хранит `declaredLength` (осевую длину), `thickness`, `height` как
`{value, status, source, note}`. Новая схема (раздел 10 требований) использует
плоские поля `lengthMm`, `thicknessMm`, `heightMm` и добавляет массив `anchors`
и отдельные объекты «размерных цепочек».
- (а) Полностью перейти на новую схему, написать миграцию из старой?
- (б) Оставить старую схему и добавить `anchors`/`dimensionChains` рядом,
      постепенно перенося?
- (в) Другой вариант?

**Ответ:** (а) — полностью перейти на новую схему, написать миграцию из
старой. localStorage бампает `schemaVersion` до v3, при загрузке v2
автоматически конвертируется.

1.2. **Идентификаторы опорных точек.**
В требованиях используется префикс `F1-A01`. У нас на F2 уже есть свои
префиксы `F2-*`. Схема идентификаторов якорей будет `{floor}-A{nn}`?
Автоматическая нумерация, gaps допустимы?

**Ответ:** да — `F1-A01`, `F1-A02`, …, `F2-A01`, … Нумерация автоматическая
через `nextId('A', ...)`. Gaps допустимы: после удаления `F1-A03` следующий
получит `F1-A04`, а не переиспользует `A03`.

1.3. **Хранение якорей.**
- (а) Массив `anchors` внутри объекта стены (как в требованиях).
- (б) Отдельный глобальный массив `state.plan.anchors` с полем `wallId`.
Первый вариант проще, второй легче для перекрёстных запросов.

**Ответ:** (а) — массив `anchors` внутри объекта стены. Меньше индексных
lookup'ов, при удалении стены якоря удаляются автоматически.

1.4. **Размерные цепочки в JSON.**
Хранить отдельно от стены (собственный массив `dimensionChains`) или как
поле стены? На стене может быть несколько цепочек (например, «по внутренней
грани» и «по осевой»)?

**Ответ:** (в) — цепочка выводится на лету из `anchors` + `dimensionReference`
стены. Отдельного объекта в JSON нет. Одна цепочка на стену. Значения
участков вычисляются из `offsetMm` соседних якорей. Множественные цепочки —
только во второй итерации при необходимости.

1.5. **Пересчёт при изменении толщины стены.**
Если толщина стены меняется, положение якорей (в мм от начала) остаётся тем
же или пересчитывается? Влияет ли толщина на видимую длину?

**Ответ:** (а) — `offsetMm` якорей и осевая длина не меняются при смене
толщины. «Внутренняя длина» пересчитывается на лету. Если ранее заявленный
`lengthMm` с dimensionReference `inside_finished_face` больше не соответствует
расчётному — валидатор покажет конфликт, пользователь сам решает.

## 2. Геометрическая интерпретация

2.1. **Что представляет линия vertex↔vertex?**
- (а) осевая (текущее);
- (б) внутренняя чистовая грань;
- (в) наружная чистовая грань.

**Ответ:** (а) осевая. SVG рисует stroke центрированно; осевая — единственный
однозначный вариант для перегородок (у них нет «наружной» стороны); не
ломает существующие координаты вершин F1/F2. Раздел 9 требований это
разрешает.

2.2. **По какой поверхности считать «общую длину стены» (`dimensionReference`)?**
- (а) осевая;
- (б) внутренняя чистовая грань;
- (в) наружная чистовая грань.

**Ответ:** по умолчанию `axis` (совпадает с геометрией). Настраивается
**поштучно на каждой стене** через инспектор. При выборе
`inside_finished_face` внутренняя длина рассчитывается как
`axial - (adj_th_v1/2) - (adj_th_v2/2)`, где `adj_th` — максимальная толщина
среди стен, сходящихся в этой вершине (кроме самой W). При заявленной длине
`inside_finished_face` редактор пересчитывает соответствующую осевую и
двигает вершину. Значение `dimensionReference` всегда видно пользователю в
инспекторе.

2.3. **Стороны углов (примыкание толстых стен).**
- (а) осевая до осевой (текущий подход);
- (б) внутренняя грань до внутренней грани;
- (в) с учётом угла и типа примыкания (T, L, X).

**Ответ:** (а) геометрия хранится по осям. При переводе в «внутреннюю
длину» применяется правило «половина максимальной толщины смежных стен» в
каждой вершине. Полноценный CSG (учёт углов, типов примыкания) не входит в
MVP — эффект от него становится заметен только при неортогональных
примыканиях.

2.4. **Толщина отдельных участков стены.**

**Ответ:** стена однородна по толщине. Утолщения (столбы, ниши) —
моделируются отдельными стенами-перегородками, примыкающими к основной.
Если понадобится «переменная толщина» — добавим позже отдельным
подобъектом `thicknessSegments`.

## 3. Опорные точки

3.1. **Автоматические точки при создании окна/двери.**
- (а) в массиве `anchors` стены как отдельные записи с `objectId`;
- (б) в самом объекте окна;
- (в) вычисляются на лету по `offsetMm` и `widthMm` окна.

**Ответ:** (а) — в `anchors` стены. Каждое окно/дверь порождает два якоря
типа `opening_start` и `opening_end` со ссылкой `objectId: <window/door.id>`.
При изменении `offsetMm` или `widthMm` объекта — оба якоря обновляются
автоматически (invariant: `A_end.offsetMm = A_start.offsetMm + width`).
При удалении окна оба якоря удаляются.

3.2. **Автоматические точки в T-соединении стен.**

**Ответ:** да — создаётся якорь типа `wall_junction` на «принимающей»
стене со ссылкой `objectId: <incoming_wall.id>`. Появляется автоматически
при создании общей вершины двух стен, где одна не заканчивается в этой
вершине (T-соединение, но не L-угол). Позволяет пользователю измерять
расстояния от угла до примыкания.

3.3. **Порядок якорей.**

**Ответ:** массив `anchors` **всегда отсортирован по `offsetMm` возрастанию**.
Инвариант поддерживается любой мутацией: `insertAnchor`, `updateOffset`,
`removeAnchor` пересортировывают массив. Первый якорь — всегда `wall_start`
(offset=0), последний — `wall_end` (offset=length).

3.4. **Якоря за пределами стены.**

**Ответ:** записать в список «Предупреждения геометрии» (например, «Окно
F1-O07 выходит за конец стены F1-W15 на 30 мм»). Не удалять автоматически
— данные остаются, пользователь сам решает: удлинить стену, сдвинуть окно
или удалить.

## 4. Режимы ввода размеров

4.1. **Режим по умолчанию для новой стены.**

**Ответ:** `chain` (размерная цепочка). Это самый интуитивный режим для
последовательных замеров рулеткой.

4.2. **Смена режима.**

**Ответ:** режим отображения переключается свободно без потери данных.
`offsetMm` каждого якоря — единственный источник правды. `mode` — поле
UI-состояния стены (не JSON модели), при переключении пересчитывается
только представление.

4.3. **Базовая точка.**

**Ответ:** MVP — (а) всегда `wall_start`. Вторая итерация — (б) выбор
любой точки через выпадающий список.

4.4. **Единица ввода.**

**Ответ:** только мм, целыми числами. Ввод дробных значений (например,
`5.4`) — принимается, округляется до целого, показывается предупреждение
«округлено до 5 мм». Ввод в других единицах (м, см) — не поддерживается,
префиксы/суффиксы не парсятся.

## 5. Проверка и конфликты

5.1. **Допуск невязки.**
- (а) 0 мм;
- (б) ±5 мм;
- (в) ±%толщины;
- (г) настраиваемый.

**Ответ:** (б) 5 мм по умолчанию + (г) `settings.tolerance.chainSumMm` в
глобальных параметрах проекта. Значение хранится в `state.plan.settings` и
редактируется через отдельную панель настроек.

5.2. **Конфликт — блокировать или разрешить с пометкой.**

**Ответ:** разрешаем сохранять с `status: "conflict"`. Красная подсветка
конфликтного значения на плане и в инспекторе, отдельная запись в списке
«Предупреждения геометрии» с указанием, какие именно значения не
согласуются и на какую величину.

5.3. **Приоритет источников при конфликте.**

**Ответ:** (а) фиксированный порядок:
`measurement > user > calculated > image`. Значение с более высоким
приоритетом считается «истинным», остальные — конфликтующими. Для MVP
timestamp не учитываем.

5.4. **Двухэтажные проверки.**

**Ответ:** опустить в MVP. Во второй итерации — явное поле
`sharedWithFloor: {floorId, wallId}` в стене; редактор проверяет, что
координаты и `lengthMm` совпадают.

## 6. Поведение при изменении размера

6.1. **Диалог выбора «что двигать».**

**Ответ:** дефолтное правило по разделу 6 требований (первый участок →
двигаем ближайший якорь, последний участок → двигаем конец стены, ширина
проёма → меняем ширину). Модальный выбор появляется только при
**`Alt+клик` по числу или `Alt+Enter` в поле ввода** — как продвинутый режим.

6.2. **Изменение длины через якорь vs через число.**

**Ответ:** MVP — только через число в правой панели. Двойной клик по
значению участка → редактирование в поле; drag якоря на плане — вторая
итерация.

6.3. **Каскадное движение смежных стен.**

**Ответ:** всегда каскадно. Общая вершина — одна для всех смежных стен.
«Разъединение» реализуется отдельной операцией «Разделить вершину» в
инспекторе вершины (не в MVP).

6.4. **Влияние на замкнутость помещения.**

**Ответ:** предупреждение в списке «Предупреждения геометрии», без
блокировки действия. Пользователь может намеренно перестраивать контур.

6.5. **Расстояние от закреплённой точки против расстояния от начала стены.**

**Ответ:** внутри модели `offsetMm` **всегда относительно `wall_start`
(v1)**. UI пересчитывает значения для отображения относительно выбранной
базовой точки. При смене базовой точки JSON не меняется, только
представление.

## 7. Интерфейс

7.1. **Панель размерной цепочки.**

**Ответ:** заменяет текущий инспектор стены в правой панели. Одна и та
же панель, разные секции: базовые свойства стены (толщина, высота, тип),
размерная цепочка списком, кнопки добавления якоря/окна/двери.

7.2. **Размерные линии на плане.**

**Ответ:** только для выделенной стены по умолчанию. Отдельный чекбокс
**«Все размеры»** в панели «Слои» — включает размерные линии для всех
стен одновременно.

7.3. **Смещение размерной линии от стены.**

**Ответ:** автоматическое — 700 мм наружу от стены по её нормали. Для
внешних стен «наружу» = за пределы здания (можно определить эвристически
по расположению относительно всех остальных стен, либо по флагу
`wall.type='exterior'`). Для перегородок — с любой стороны, выбирается
детерминированно. Ручная настройка смещения — не в MVP.

7.4. **Стиль размерных выносок.**

**Ответ:** упрощённый в MVP — число рядом с участком, короткая
размерная линия с засечками на концах. Полноценные архитектурные выноски
(тонкие выносные, засечки по нормали к размерной линии, шрифт по ГОСТ) —
позже.

7.5. **Редактирование числа inline.**

**Ответ:** только в панели справа в MVP. Inline-редактирование на плане
(HTML `foreignObject` или overlay) — вторая итерация.

7.6. **Отображение статусов (6 значений).**

**Ответ:** да, шесть статусов:
- `unknown` — серый;
- `estimated` — жёлтый;
- `entered` — оливковый;
- `calculated` — синий (**новый**);
- `verified` — зелёный;
- `conflict` — красный.

7.7. **Мобильная работа.**

**Ответ:** не в MVP. Тач-жесты и мобильная вёрстка — после стабилизации
desktop-версии.

## 8. Двери и окна

8.1. **Совместимость с текущей моделью двери/окна.**

**Ответ:** плоские числа. Каждое размерное поле — пара
`fieldMm: number` + `fieldStatus: enum`. Например,
`{ offsetMm: 1200, offsetStatus: "entered", widthMm: 1400, widthStatus:
"measurement" }`. Отдельного объекта-обёртки нет. Дополнительные метаданные
(source, note) — в отдельном подобъекте `provenance` при необходимости.

8.2. **Ширина как отдельный участок размерной цепочки.**

**Ответ:** генерируется из объекта окна автоматически. `widthMm` окна —
единственный источник правды. В размерной цепочке участок между
`opening_start` и `opening_end` подписывается «Ширина окна» и его значение
= `window.widthMm`. Правка ширины в цепочке — обновляет `window.widthMm`.

8.3. **Перенос окна на другую стену.**

**Ответ:** только через выпадающий список «Родительская стена» в
инспекторе, как сейчас. Drag-and-drop между стенами — позже (сложно
реализовать корректно с автоматическим пересчётом offsetMm).

8.4. **Соединения (двери в перегородках между двумя комнатами).**

**Ответ:** не хранить явно, какая сторона «внутренняя» относительно
комнаты. Текущее `swing: in/out` относительно нормали стены — достаточно.
Если позже понадобится связка «комната откуда/куда открывается» — добавим
поля `fromRoomId` / `toRoomId`.

## 9. Работа с существующими данными

9.1. **Импорт текущего plan-f1/plan-f2 в новую модель.**

**Ответ:** автоматически при первой загрузке v2 → v3. Функция миграции:
- для каждой стены создаёт `anchors: [wall_start, wall_end]`;
- для каждой двери/окна создаёт два якоря `opening_start` и `opening_end`
  из `distance.value` и `distance.value + width.value`;
- измерения статуса перекладываются в новые поля (`declaredLength` →
  `lengthMm` + `lengthStatus`, и т.д.).

9.2. **Стены с толщиной по умолчанию (400/150 мм, `estimated`).**

**Ответ:** миграция ставит `dimensionReference: "axis"` для всех стен —
значение сохраняется «как было», не пересчитывается. Пользователь потом
сам решает, какие стены переключить на `inside_finished_face`.

9.3. **Отсутствующие цепочки.**

**Ответ:** генерируются автоматически при миграции. Якоря создаются из
существующих дверей/окон; для стен без проёмов — только
`wall_start` и `wall_end`. Дополнительно, T-соединения детектируются и
создают якоря `wall_junction`.

## 10. Приоритет и фазирование

10.1. **MVP-объём.**

**Ответ:** 7 пунктов:
1. Выбор стены на плане.
2. Автоматические якоря `wall_start`, `wall_end`, `opening_start`,
   `opening_end`, `wall_junction`.
3. Редактирование участков цепочки числом в панели инспектора.
4. Автопересчёт общей длины стены.
5. Проверка суммы участков с цветовой индикацией (6 статусов).
6. Сохранение/загрузка нового JSON (schemaVersion v3).
7. Undo/redo для правок якорей и размеров.

Опущено: пользовательские точки, режим от базовой точки, закрепление
произвольной точки, inline-редактирование числа на плане, полноценные
архитектурные размерные выноски, двухэтажные проверки, диалог
«что двигать» (только Alt-модификатор).

10.2. **Что после MVP.**

**Ответ:** приоритеты второй итерации:
1. Пользовательские точки на стене (колонны, ниши, произвольные
   ориентиры).
2. Режим измерений от базовой точки (4.3 → выбор точки).
3. Drag якорей на плане (визуальное перемещение окна/двери).
4. Диалог выбора «что двигать» (Alt-модификатор → полноценный интерфейс).
5. Inline-редактирование числа на плане (двойной клик по значению).

Третья итерация:
6. Полноценные размерные выноски по ГОСТ.
7. Двухэтажные проверки (`sharedWithFloor`).
8. Экспорт в SVG/PDF.

10.3. **Сроки/итерации.**

**Ответ:** 3 итерации:
- **Итерация 1 — модель и миграция** (~1 неделя):
  миграция v2 → v3, структура якорей, инварианты, undo/redo для новой
  модели, без изменений в UI.
- **Итерация 2 — UI цепочек** (~1 неделя):
  инспектор размерной цепочки, редактирование участков, визуализация
  якорей на плане, простые размерные линии.
- **Итерация 3 — проверки и полировка** (~3–5 дней):
  расчёт конфликтов, приоритеты источников, `dimensionReference`
  переключение, автопересчёт «внутренней длины».

## 11. Смежные вопросы

11.1. **Единицы измерения.**

**Ответ:** только мм. Настройка `state.plan.units` фиксирована = `"mm"`.

11.2. **Экспорт размерной цепочки.**

**Ответ:** не в MVP. Экспорт JSON (в новой схеме v3) остаётся; SVG/PDF —
третья итерация.

11.3. **История правок (undo/redo).**

**Ответ:** батчить по «пользовательской операции». Одна правка одного
поля = одна запись в истории (даже если она физически изменила два якоря —
например, `opening_start` и `opening_end` при изменении ширины окна).
Ctrl+Z откатывает пользовательское действие, а не отдельные внутренние
мутации.

11.4. **Многопользовательское редактирование.**

**Ответ:** один пользователь на вкладку, локальный `localStorage`. Без
CRDT, без синхронизации между вкладками. Экспорт JSON остаётся способом
