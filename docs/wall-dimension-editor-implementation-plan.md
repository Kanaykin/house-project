# План реализации редактора размеров стен (step-by-step)

Основано на ответах из
[wall-dimension-editor-questions.md](wall-dimension-editor-questions.md) и
требованиях из
[wall-dimension-editor-requirements.md](wall-dimension-editor-requirements.md).

Разделено на 3 итерации. Каждая итерация — отдельный набор PR-ов, тестируется
целиком, оставляет редактор в работоспособном состоянии.

---

## Итерация 1 — Модель данных и миграция

**Цель:** перейти на новую JSON-схему v3, автоматически конвертировать
существующие данные, обеспечить инварианты для якорей. UI пока не меняется —
работают все текущие функции, просто на новых структурах.

**Ожидаемый срок:** ~1 неделя.

### Шаг 1.1. Определить структуры JSON v3

**Файлы:** `docs/wall-dimension-editor-requirements.md` (уже есть),
`house-project/app.js` (обновить константы схемы).

**Действия:**
1. Зафиксировать v3-структуру плана в комментарии в начале `app.js`.
2. Определить типы якорей: `wall_start`, `wall_end`, `opening_start`,
   `opening_end`, `wall_junction`, `user_point`.
3. Определить набор `dimensionReference`: `axis` (по умолчанию),
   `inside_finished_face`, `outside_finished_face`, `structural_face`.
4. Определить набор статусов размеров (6 значений).
5. Определить набор источников: `image`, `user`, `measurement`, `calculated`,
   `imported`.

**Критерий готовности:** в начале `app.js` есть закомментированный JSDoc-блок
с типами структур. `SCHEMA_VERSION = '3.0'`.

### Шаг 1.2. Реализовать функцию миграции v2 → v3

**Файлы:** `house-project/app.js` — новая функция `migrateV2toV3(oldPlan)`.

**Действия:**
1. Для каждой стены:
   - `declaredLength.value` → `lengthMm` (число), `declaredLength.status` →
     `lengthStatus`.
   - `thickness.value` → `thicknessMm`, `thickness.status` → `thicknessStatus`.
   - `height.value` → `heightMm`, `height.status` → `heightStatus`.
   - Добавить `dimensionReference: "axis"`.
   - Добавить пустой массив `anchors: []`.
   - Добавить `fixedAnchorId: null` (по умолчанию — начало стены).
2. Для каждого окна:
   - `distance.value` → `offsetMm`, `distance.status` → `offsetStatus`.
   - Аналогично для `width`, `height`, `sillHeight`.
3. Для каждой двери — аналогично окну.
4. **Сгенерировать якоря** для каждой стены:
   - `wall_start` (offsetMm=0).
   - `wall_end` (offsetMm=lengthMm или axial-length по геометрии, если
     `declaredLength` не задан).
   - Для каждого окна/двери на этой стене: `opening_start` (offsetMm =
     opening.offsetMm), `opening_end` (offsetMm = opening.offsetMm +
     opening.widthMm), с полем `objectId: <opening.id>`.
5. **Детектировать T-соединения:** если вершина стены W1 совпадает с
   вершиной внутри геометрии стены W2 (не концом), добавить якорь
   `wall_junction` на W2 с `objectId: <W1.id>`.
6. Отсортировать `anchors` каждой стены по `offsetMm`.
7. Проставить `provenance` для якорей: source = `image` (наследуется от
   миграции старых данных).

**Критерий готовности:** тест на существующих `plan-f1.json` / `plan-f2.json`
показывает корректную конверсию: якоря создаются для каждой стены и
проёма, длины совпадают, кол-во элементов сохраняется.

### Шаг 1.3. Обновить `loadLocal` / `saveLocal` / `buildInitialFloors`

**Файлы:** `house-project/app.js`.

**Действия:**
1. Поднять ключ `STORAGE_KEY` до `planhouse.v3.state` (старый ключ v2
   остаётся, чтобы можно было прочитать и мигрировать).
2. `loadLocal()`:
   - Пробовать v3-ключ. Если есть — грузить как v3.
   - Если нет — пробовать v2-ключ. Если есть — прогнать через
     `migrateV2toV3`, сохранить как v3, удалить v2.
   - Если ни то, ни другое — идти в `buildInitialFloors`.
3. `buildInitialFloors()`:
   - Конвертировать `SOURCE_PLANS.F1` / `F2` из source-схемы сразу в
     v3-структуру (через `convertSourceToEditor` + `migrateV2toV3`, либо
     переписать `convertSourceToEditor` сразу в v3).
4. `saveLocal()`: всегда пишет v3.

**Критерий готовности:** открыть редактор после апдейта — данные загружаются
корректно без вмешательства пользователя. Открытие в чистом браузере —
загружает source-планы и сразу конвертирует в v3.

### Шаг 1.4. API работы с якорями

**Файлы:** `house-project/app.js` — новые функции.

**Действия:**
1. `getAnchors(wall)` — возвращает отсортированный массив якорей.
2. `insertAnchor(wall, anchor)` — вставляет якорь с соблюдением сортировки,
   возвращает его ID.
3. `updateAnchorOffset(wall, anchorId, newOffsetMm)` — меняет offset и
   пересортировывает массив. Валидирует, что offset ∈ [0, wall.lengthMm].
4. `removeAnchor(wall, anchorId)` — удаляет якорь. Запрет удаления
   `wall_start` / `wall_end`.
5. `nextAnchorId(state)` — генерация ID `{floor}-A{nn}` по текущему этажу.
6. `syncAnchorsForOpening(wall, opening)` — вызывается при изменении
   `offsetMm` или `widthMm` окна/двери: обновляет два соответствующих
   якоря `opening_start` и `opening_end`.
7. `syncAnchorsForJunction(wall1, wall2)` — вызывается при изменении общей
   вершины: пересоздаёт `wall_junction` на «принимающей» стене.

**Критерий готовности:** unit-тесты (можно inline в комментариях) для
каждой функции с несколькими сценариями.

**Файлы:** `house-project/app.js` — функции `renderWallInspector`,

**Действия:**

4. Импорт/экспорт JSON — теперь работает с v3-схемой.
5. Функция `applyDeclaredLength` — заменяется на
   `applyDeclaredLengthByReference(wall)`: учитывает `dimensionReference`
   и пересчитывает вершину.

**Критерий готовности:** все текущие функции (клик по стене / окну /
двери, редактирование длины, drag вершины, drag окна, undo/redo,
экспорт/импорт) работают на новой схеме. Визуальных отличий пользователь
не замечает — это чисто под-капотный рефактор.

### Шаг 1.6. Undo/redo для новой модели

**Файлы:** `house-project/app.js` — `pushHistory`, `undo`, `redo`.

**Действия:**
1. Проверить, что снапшот `state.plan` сохраняет якоря корректно.
2. Ввести концепцию «пользовательской операции»: одна запись в истории
   может содержать несколько атомарных мутаций. Функция `withTransaction(fn)`
   оборачивает несколько правок в одну history-entry.
3. При изменении ширины окна: одна операция → две мутации (два якоря
   `opening_start` и `opening_end` обновляются) → одна запись undo.

**Критерий готовности:** Ctrl+Z откатывает целые операции, а не отдельные
поля. Test-case: изменил ширину окна → Ctrl+Z → всё вернулось.

## Итерация 2 — UI размерной цепочки

**Цель:** пользователь видит и редактирует размерную цепочку выделенной
стены в правой панели, размерные линии рисуются на плане.

**Ожидаемый срок:** ~1 неделя.

### Шаг 2.1. Секция размерной цепочки в инспекторе стены

**Файлы:** `house-project/app.js` — `renderWallInspector`.

**Действия:**
1. Под базовыми полями стены (длина, толщина, высота, тип) добавить
   секцию **«Размерная цепочка»**.
2. Прочитать якоря стены, построить массив участков: пары соседних якорей
   → участок с длиной = `next.offsetMm - prev.offsetMm`.
3. Для каждого участка вывести строку в списке:
   ```
   1. Угол → окно F1-O01     [1200 мм ▼]  🟡
   2. Ширина окна F1-O01     [1400 мм]    🔵
   3. Окно F1-O01 → угол     [2350 мм ▼]  ⚪
   ```
   где значок статуса — цветной кружок (см. 7.6).
4. Ввод в поле обновляет:
   - Участок между `wall_start` и `opening_start` → двигаем окно
     (правило 6.1).
   - Участок «ширина окна» → меняет `window.widthMm`, оба якоря
     `opening_*` синхронизируются.
   - Участок между последним якорем и `wall_end` → меняет `wall.lengthMm`
     (перемещаем вершину v2 вдоль оси).
5. Выпадающая иконка ▼ рядом со значением открывает мини-меню:
   «изменить статус», «удалить связанный якорь», «пометить как
   verified».
6. Внизу секции — итог: **«Итого по осям: 4950 мм»** и, если
   `dimensionReference != "axis"`, дополнительно
   **«По внутренней грани: 4550 мм»**.

**Критерий готовности:** выделяя стену, вижу список её участков с
цифрами, могу править каждое число, изменения синхронизируются с
геометрией.

---

### Шаг 2.2. Кнопки добавления в цепочку

**Действия:**
1. **[+ Точка]** — открывает мини-диалог: тип (`user_point`), подпись,
   offsetMm от начала. Создаёт якорь.
2. **[+ Окно на этой стене]** — создаёт окно с шириной 900, offsetMm по
   центру стены, порождает якоря автоматически.
3. **[+ Дверь на этой стене]** — аналогично.

**Критерий готовности:** можно добавить пользовательскую точку прямо из
инспектора стены; появляется новая строка в цепочке.

---

### Шаг 2.3. Визуализация якорей на плане

**Файлы:** `house-project/app.js` — новый `renderAnchors()`.

**Действия:**
1. Для выделенной стены нарисовать все якоря маленькими значками:
   - `wall_start` / `wall_end` — синие треугольники.
   - `opening_start` / `opening_end` — оранжевые ромбы.
   - `wall_junction` — жёлтый круг.
   - `user_point` — фиолетовый квадрат.
2. Значки перпендикулярны стене (нормаль от осевой на 200 мм наружу).
3. При клике по значку — подсветить связанную строку в инспекторе.

**Критерий готовности:** на выделенной стене видно все якоря, каждый
подписан ID.

---

### Шаг 2.4. Размерные линии на плане (упрощённый стиль)

**Файлы:** `house-project/app.js` — новый `renderDimensionLines()`.

**Действия:**
1. Для выделенной стены нарисовать одну размерную линию, параллельную
   стене, на расстоянии 700 мм по нормали.
2. Между каждой парой соседних якорей на этой линии — короткие
   перпендикулярные засечки.
3. Между засечками — текст с числом (значение участка в мм).
4. Цвет числа = цвет статуса участка.
5. Если участок — окно/дверь, дополнительно подписать выше:
   «W1400» (ширина окна) или «D900».

**Критерий готовности:** визуально на плане у выделенной стены появляется
размерная линия с числами всех участков.

---

### Шаг 2.5. Чекбокс «Показать все размеры» в панели «Слои»

**Действия:**
1. Добавить в панель «Слои» чекбокс.
2. Когда включён — `renderDimensionLines()` рендерит линии для ВСЕХ стен,
   не только для выделенной.

**Критерий готовности:** чекбокс работает, план не тормозит при 50+
стенах.

---

### Шаг 2.6. Изменение `dimensionReference` в UI

**Действия:**
1. В инспекторе стены — выпадающий список **«Опорная поверхность»** с
   вариантами `axis` / `inside_finished_face` / `outside_finished_face`.
2. При смене:
   - Если `axis` — цепочка отображается «как есть» (offsetMm напрямую).
   - Если `inside_finished_face` — участки пересчитываются на лету:
     от значений вычитаются полутолщины смежных стен в вершинах.
3. Итоговое число «По внутренней грани» становится основным, «По осям»
   — вторичным в hint.

**Критерий готовности:** переключение показывает разные числа в цепочке
БЕЗ изменения геометрии.

## Итерация 3 — Проверки, конфликты, полировка

**Цель:** система знает о конфликтах, показывает их, помогает разрешить.

**Ожидаемый срок:** ~3–5 дней.

### Шаг 3.1. Автопересчёт с учётом `dimensionReference`

**Действия:**
1. При вводе внутренней длины стены — вычислить требуемую осевую и
   передвинуть вершину v2.
2. При изменении толщины стены — пересчитать «внутреннюю» интерпретацию
   всех связанных величин.
3. Пометить пересчитанные значения статусом `calculated`.

**Критерий готовности:** ввод внутренней длины «5000 мм» на стене с
толщиной 400 (внутренние стены рядом 150) → осевая длина ставится
5075, `calculated`-статус подсвечен.

---

### Шаг 3.2. Проверка суммы цепочки

**Действия:**
1. Функция `checkChainSum(wall)` — сравнивает сумму всех участков с
   `wall.lengthMm`.
2. Допуск = `state.plan.settings.tolerance.chainSumMm` (по умолчанию 5
   мм).
3. При превышении: помечает `wall.lengthStatus = "conflict"` и записывает
   в список «Предупреждения геометрии».

**Критерий готовности:** ручной сбой (ввод неверного участка) —
подсвечивает конфликт красным, показывает величину невязки.

---

### Шаг 3.3. Панель «Настройки проекта»

**Файлы:** новая секция в `index.html` + обработчики в `app.js`.

**Действия:**
1. Кнопка «⚙ Настройки» в верхней панели.
2. Открывает выпадающую панель / модалку с полями:
   - Допуск невязки цепочки (мм) — по умолчанию 5.
   - Высота потолка по умолчанию (мм).
   - Единицы (только мм, disabled в MVP).
3. Сохранение в `state.plan.settings`.

**Критерий готовности:** настройки сохраняются в JSON и восстанавливаются
при перезагрузке.

---

### Шаг 3.4. Alt+клик для диалога «что двигать»

**Действия:**
1. При `Alt+клик` по значению участка — открывается мини-диалог с
   радио-кнопками:
   - Переместить окно (если применимо).
   - Переместить конец стены.
   - Изменить соседний участок.
2. Выбор применяется вместо дефолтного правила.

**Критерий готовности:** обычный клик → дефолтное поведение, Alt+клик →
выбор действия.

---

### Шаг 3.5. Приоритет источников при конфликте

**Действия:**
1. Функция `resolveConflict(wall, chainSegment)` — если два источника
   дают разные значения для одного участка, выбирает победителя по
   приоритету `measurement > user > calculated > image`.
2. Проигравшие значения помечаются `conflict`, победитель — сохраняет
   свой статус.
3. В UI — при клике на участок с конфликтом показывается список всех
   значений и их источников.

**Критерий готовности:** сценарий: пользователь ввёл ручное значение
поверх calculated → предыдущее calculated помечено как конфликт.

---

### Шаг 3.6. Документация

**Файлы:** обновить `house-project/README.md`, добавить раздел про
размерные цепочки.

**Действия:**
1. Как работать с размерной цепочкой (пошагово с картинкой).
2. Как выбрать `dimensionReference`.
3. Что значит каждый цвет статуса.
4. Как читать предупреждения.

**Критерий готовности:** новый пользователь может пройти сценарий
«ввести стену 5м с окном 1.4м посередине» без обращения к разработчику.

## Обзор этажей и критерии полной готовности

Функция считается готовой (по требованиям, раздел 15), если пользователь
может:

1. ✅ Выбрать стену на плане.
2. ✅ Указать базовую точку (в MVP всегда `wall_start`).
3. ✅ Добавить окно или дверь на стену.
4. ✅ Ввести расстояние от угла до проёма.
5. ✅ Ввести ширину проёма.
6. ✅ Ввести расстояние от проёма до следующего угла.
7. ✅ Получить общую длину стены автоматически.
8. ✅ Увидеть расхождение при конфликте размеров.
9. ⚠ Выбрать, какой объект перемещать при изменении размера — только
   через `Alt+клик` в MVP.
10. ✅ Сохранить все точки, объекты, размеры и статусы в JSON v3.

## Порядок выполнения

Рекомендую **не смешивать итерации**:

- Сначала полностью закрыть Итерацию 1 (модель + миграция), убедиться, что
  ничего не сломано.
- Только потом Итерация 2 (UI цепочек).
- Итерация 3 добавляется поверх, когда UI цепочек работает и есть с чем
  проверять конфликты.

Промежуточные PR-ы можно делать чаще — например:
- PR 1: `migrateV2toV3` + типы (Шаги 1.1–1.2).
- PR 2: `loadLocal` v3 + API якорей (Шаги 1.3–1.4).
- PR 3: инспекторы и рендер под v3 (Шаги 1.5–1.7).
- PR 4: инспектор цепочки (Шаг 2.1).
- PR 5: кнопки добавления + якоря на плане (Шаги 2.2–2.3).
- PR 6: размерные линии + чекбокс «все» (Шаги 2.4–2.5).

## Риски и что учесть

1. **Миграция v2 → v3 необратима.** Пользователь потеряет доступ к
   старому формату. → Экспорт-точка перед миграцией (кнопка «Backup v2»),
   или сохранение старого JSON в отдельный ключ.
2. **Автогенерация якорей `wall_junction` в T-соединениях.** Может
   создать много якорей на длинной стене. → Ограничить: только для
   соседних стен, не создавать якорь если он ближе 100 мм к концу стены.
3. **Пересчёт «внутренней длины» при переменных толщинах смежных стен.**
   Требует консистентности между стенами. → Инвариант: при изменении
   толщины стены W пересчитать `computed inside length` для всех стен,
   которые упираются в W.
4. **Undo/redo с транзакциями.** Легко ошибиться и получить рваную
   историю. → Автотест: пройти 20 случайных операций, потом Ctrl+Z 20
   раз, проверить что plan вернулся к исходному.
5. **Кол-во якорей может расти быстро** (окно + T-соединения). Для стены
   с 3 окнами и 2 T-примыканиями — 4 + 6 + 2 = 12 якорей. → Пока не
   критично; если станет — оптимизировать рендер якорей.
